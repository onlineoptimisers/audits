Module 02Multi-Agent Orchestration

Single-threaded Claude is powerful. Multi-agent Claude is a team. You're not waiting for one task to finish before starting the next — you're running 5, 10, or 13 agents in parallel, each owning a different piece of the work.

What Agents Are

An agent is a Claude instance that gets its own context, its own task, and runs independently. The main Claude (the one you're talking to) acts as the orchestrator — it decides what agents to spawn, what each one works on, and integrates their outputs when they're done.

ConceptWhat It Means
OrchestratorThe main Claude you talk to. Plans, delegates, integrates.
SubagentA spawned Claude that works on one specific task. Returns results to orchestrator.
Background agentA subagent that runs without blocking. You get notified when it's done.
Foreground agentA subagent that blocks until complete. Use when you need results before next step.
Worktree agentAn agent that works on an isolated git branch. No file conflicts.

Parallel Orchestration Patterns

Pattern 1: Fan-Out / Fan-In

Spawn N agents for independent tasks, wait for all to return, integrate results.

I need 9 client audits done. Spawn one agent per client. Each agent reads the client profile at knowledge/clients/[slug]/profile.md and runs a full SEO + AI visibility check. Save results to deliverables/[slug]/. Go.

Pattern 2: Pipeline

Agent A's output feeds Agent B, which feeds Agent C. Sequential but each agent is specialised.

Agent 1: scrape the prospect's website with Firecrawl. Agent 2: take that data and run the AI visibility audit. Agent 3: take the audit and build the HTML presentation deck. Pipeline it.

Pattern 3: Explore + Decide + Execute

Fast cheap agents explore (Haiku), orchestrator decides (Sonnet/Opus), execution agents build (Sonnet).

Model routingExplore agents = Haiku (fast, cheap, 10+ parallel) Decide = Sonnet (judgment, architecture) Execute agents = Sonnet (writing, building) Plan = Opus (deep reasoning, strategy)

Plow Mode — Burning Credits Efficiently

When you have compute budget to burn and a backlog of unblocked work, plow mode maximises parallel output:

Plow Mode Rules
  1. Each agent gets its own output file or directory. No two agents write to the same path. Collisions kill throughput.
  2. Main Claude owns shared surfaces — memory files, master plans, git commits, deliverables index.
  3. Hand each agent an exhaustive input file list. Don't make agents grep for files. Give them the exact paths.
  4. Set hard time-stops per agent (45-90 min). Prevents runaway sessions.
  5. Commit per agent, not per batch. Each agent's output gets its own commit.
Real example: 5 parallel Opus agents, one session

Module 2 Checkpoint

  • You can describe 3 parallel patterns (fan-out, pipeline, explore/decide/execute)
  • You know the plow mode rules (unique targets, commit per agent, hand over file lists)

Module 03The Skills Factory

Skills turn repeatable tasks into one-command workflows. Instead of explaining what you want every time, you type /audit or /monthly-report and Claude runs the whole playbook. A mature workspace has 30-55+ skills covering every recurring task.

Skill Anatomy

.claude/skills/quick-audit.md--- name: quick-audit description: Run a prospect AI + SEO visibility audit model: claude-haiku-4-5-20251001 --- # Quick Audit Skill When invoked with /quick-audit [business-name] [city]: ## Step 1: Gather Data - Search for the business website - Check Google Business Profile (reviews, rating) - Test schema markup presence - Run DataForSEO ranked keywords check ## Step 2: AI Visibility Test - Query ChatGPT: "[service] in [city]" — is business cited? - Query Perplexity: same query — is business cited? - Score: X/6 citation slots filled ## Step 3: Generate Report - Build markdown report at deliverables/[slug]/quick-audit.md - Include: top 3 quick wins, competitor comparison, score ## Step 4: Build HTML Deck - Use template at templates/presentations/ai-audit-deck-template.html - Replace placeholders with real data - Save to deliverables/[slug]/lead-audit/quick-audit-v1.html ## Step 5: Deploy - Copy to audits/ GitHub Pages repo - git add, commit, push - Return live URL

The Frontmatter

FieldWhat It Does
nameThe slash command trigger: /quick-audit
descriptionShows when browsing available skills with /
modelWhich model runs the skill. Haiku = cheap bulk. Sonnet = judgment. Opus = deep reasoning.

Chaining Skills

Skills can call other skills. A monthly report skill might internally call the data-pull skill, then the format skill, then the delivery skill:

Chained workflow/monthly-report elite-movers → internally runs: data pull from Sheets → internally runs: AI visibility recheck → internally runs: format as Google Slides template → internally runs: draft delivery email → outputs: report + email + Drive link

Model Routing in Skills

Not every skill needs the same brain. Route by cost and complexity:

Skill TypeModelWhy
Data extraction, formatting, cleanupHaikuFast, cheap, no judgment needed
Audits, reports, client-facing contentSonnetNeeds quality judgment
Strategy, architecture, complex planningOpusDeep reasoning required
Batch transforms (100+ items)HaikuCost control at scale
The Orchestrator Paradox
The model in the frontmatter is the execution model, not the writing model. You always build skills using Sonnet/Opus (you need judgment to write the workflow). But the skill runs on whatever model is in the frontmatter. Write once with the big brain, run many times with the cheap brain.
Module 3 Checkpoint
  • You can write a skill with frontmatter + multi-step workflow
  • You understand model routing (Haiku for bulk, Sonnet for judgment, Opus for planning)
  • You know skills can chain other skills

Module 04Autonomous Mode — Loops, Background Agents, Monitoring

This is where Claude goes from "tool you use" to "system that runs." Autonomous workflows execute without you watching, report back when done, and can run on schedules.

Loops (/loop)

The /loop command tells Claude to repeat a task on a schedule. Claude self-paces: it picks the right delay between iterations based on what it's waiting for.

/loop Check the build status every 2 minutes until it passes, then deploy to production.
/loop Monitor the audits/ GitHub Pages deploy. When the page returns HTTP 200, notify me and stop.

Claude manages its own sleep intervals — shorter when actively waiting (60-270s to stay in cache), longer when idle (20-30 min).

Background Agents

Background agents run without blocking your main conversation. You kick them off, keep working, and get notified when they finish.

How it worksYou: "Build the pitch library in the background while I work on the master plan." Claude: [spawns background agent with the pitch library task] Claude: [continues working with you on the master plan] ... [notification]: Background agent completed. Output: docs/pitch-library.md (1,443 lines)

Monitoring Patterns

Heartbeat Monitoring

For agent systems with many components, build a heartbeat monitor:

heartbeat.py (concept)# Every agent writes a timestamp to its Airtable record # Monitor checks: if last_heartbeat > 5 min ago, alert # # Real example: 36 agents, 255 Airtable records, # 10/10 guardrails, heartbeat checks every 60s

Self-Healing Patterns

  • API fallback chains: If OpenAI is down, fall back to Perplexity. If Perplexity is down, fall back to web search.
  • Kill switches: If error rate > 10% in 5 minutes, pause all automated operations and notify.
  • Graceful degradation: If premium data source fails, produce the report with whatever data is available, clearly marking gaps.
Safety Rule
No recurring crons or automated scrapes until you've audited the credit cost. All automation starts as manual-trigger only. Promote to scheduled only after you've seen the per-run cost.

Module 05The Memory Architecture

Memory is what makes Claude feel like a partner, not a tool. Across hundreds of sessions, Claude remembers your preferences, your clients, your decisions, your project state — all stored as small files that get loaded at the start of every conversation.

The 4 Memory Types

TypeWhat It StoresExample
UserWho you are, your role, preferences"Donal is an agency owner, prefers terse responses, British English"
FeedbackWhat to do / not do (corrections + confirmations)"Never mock the database in integration tests"
ProjectActive work, decisions, deadlines"Merge freeze starts 2026-03-05 for mobile release"
ReferenceWhere to find things in external systems"Pipeline bugs tracked in Linear project INGEST"

MEMORY.md — The Index

MEMORY.md is loaded into every conversation. It's an index, not content — each entry links to a detailed file. Keep it under 200 lines (anything after gets truncated).

MEMORY.md structure# Memory Index ## User - [user_profile.md](user_profile.md) — Agency owner, prefers terse, British English ## Feedback - [feedback_git.md](feedback_git.md) — Alex owns all git decisions, never ask Donal - [feedback_testing.md](feedback_testing.md) — Integration tests hit real DB, not mocks ## Project - [project_elevare.md](project_elevare.md) — Tutoring business build, DNS blocked on domain - [project_dashboard.md](project_dashboard.md) — Ops dashboard V1 live on CF Pages ## Reference - [reference_apis.md](reference_apis.md) — All API keys + MCP status tracker

Collaboration Zones

When working with multiple people or projects, memory gets scoped into zones:

ZoneWho Sees ItWhat's In It
SoloYou + Claude onlyBusiness strategy, financials, personal goals
Partner-sharedYou + specific collaboratorShared project state, decisions, deliverables
Client-facingAnyone who reads the outputOnly professional, scrubbed content
Memory vs CLAUDE.md vs Plans
CLAUDE.md = permanent identity (rarely changes). Memory = evolving context (updates weekly). Plans = current task (changes per session). Don't save task-specific details to memory — those belong in the plan.

Module 06MCP Ecosystem — Connecting Everything

MCP (Model Context Protocol) is Claude Code's native plugin system. Each MCP server gives Claude a new set of tools — read Google Sheets, search Drive, send Slack messages, fetch web pages.

What MCP Unlocks

Without MCP, Claude reads and writes files in your workspace. With MCP:

  • Claude can read your Google Sheets without you pasting data
  • Claude can search your Drive by filename and read documents
  • Claude can fetch any web page and extract structured data
  • Claude can authenticate with OAuth to services on your behalf

The Full MCP Stack

MCP ServerProviderWhat It Does
Google DriveAnthropic (built-in)Read/search Drive files, Sheets, Docs
Google SheetsAnthropic (built-in)Read/write spreadsheet cells and ranges
Web FetchBuilt-inFetch any URL, extract content
Web SearchBuilt-inSearch the internet in real-time
FirecrawlCommunityDeep site crawls, structured extraction
SlackCommunityRead/send messages to channels
GitHubCommunityIssues, PRs, code search via API
Setting up MCP servers

MCP servers are configured in .claude/settings.json or via Claude Code's settings UI. Each server typically needs:

  1. An npm package or local script
  2. Authentication (OAuth flow or API key in env var)
  3. A one-time approval when Claude first uses it

Ask Claude: "Set up the Google Drive MCP server" and it'll walk you through the process.

MCP Security
Only connect MCP servers from verified providers. Anthropic does not audit third-party MCP servers. Review each one before connecting — treat it like installing software. Never let an MCP server access financial data without explicit approval.

Module 08The Frontier — Agent SDK, Swarms, What's Next

Everything above is what's working today. Here's what's coming — and how to position yourself to use it the moment it arrives.

The Claude Agent SDK

Anthropic's Agent SDK lets you build custom agents that run outside of Claude Code. Think of it as Claude-as-a-library — you write Python or TypeScript, call Claude's API, and build agents that:

  • Run on your own server (VPS, cloud function, local machine)
  • Have custom tools (your own APIs, databases, services)
  • Execute multi-step workflows autonomously
  • Hand off to each other (Agent A completes → triggers Agent B)
Agent SDK concept (Python)from anthropic import Agent audit_agent = Agent( model="claude-sonnet-4-6", tools=[dataforseo_tool, firecrawl_tool, sheets_tool], system_prompt="You are an SEO audit specialist..." ) result = audit_agent.run( "Audit example-business.com for AI visibility" )

Swarms — Agents That Feed Each Other

A swarm is the end state: a network of agents where each agent's output is another agent's input. No human in the loop for routine operations.

Swarm architecture (vision)Lead Scraper Agent ↓ finds prospect Audit Agent ↓ runs AI visibility check Pitch Builder Agent ↓ creates custom deck Email Drafter Agent ↓ writes personalised outreach Campaign Agent ↓ loads into Instantly Follow-Up Agent ↓ monitors replies, escalates hot leads Calendar Agent ↓ books the call Pre-Call Brief Agent ↓ prepares you for the meeting You show up. You close.
The 80/20 of Swarms
You don't need the full swarm to get value. Build one 3-agent pipeline that handles your most common workflow end-to-end. Most people will never build past 3 connected agents — and that's enough to 10x their output.
